## 1. 현재 상태 진단

| 항목 | 현재 상태 | 평가 |
|------|----------|------|
| 프레임워크 | Next.js 16 (App Router) | 최신, 적절 |
| DB | Supabase PostgreSQL + Prisma | 적절 |
| 인증 | NextAuth Credentials (JWT) | 기능적이나 소셜 로그인 부재 |

| 데이터 소스 | 12개 HIGH RISK 파일 (스크래핑) | 상용화 차단 요소 |

### 1.3 핵심 병목 (Critical Bottlenecks)

1. **Vercel 타임아웃**: 보고서 생성 60초+ 소요, Serverless 1분 제한 초과 위험
2. **Serverless 캐시 비효율**: `MemoryCache`가 cold start마다 초기화됨
3. **데이터 소스 법적 리스크**: 12개 파일이 웹 스크래핑으로 상용화 불가
4. **단일 인증 방식**: Credentials만 지원, 소셜 로그인 없음
5. **API 비용 선형 증가**: 사용자 수 비례로 OpenAI 비용 증가 (캐싱/배치 미적용)

## 2. Vercel 타임아웃 최적화 방안

### 2.1 문제 분석

```
현재 보고서 생성 흐름:
POST /api/reports/daily/generate
  -> collectReportData() [20-30초: 10종목 x 6-8 API 호출]
  -> analyzeReportData() [40-60초: 13건 GPT 호출 직렬]
  -> buildReportMeta()   [<1초]
  -> DB upsert           [<1초]
총: 60-90초 (Vercel 타임아웃 60초 초과)
```

### 2.2 해결 방안 (우선순위순)

#### 방안 A: Streaming + 단계별 응답 (권장, 즉시 적용 가능)

```
흐름:
1. POST 요청 -> 즉시 202 Accepted + reportId 반환
2. 데이터 수집 완료 -> DB에 중간 상태 저장 (status: "collecting")
3. AI 분석 시작 -> 종목별로 완료될 때마다 DB 업데이트
4. 프론트엔드: polling (GET /api/reports/{id}/status) 으로 진행률 표시
5. 전체 완료 -> status: "completed"
```

| 항목 | 세부 |
|------|------|
| 구현 난이도 | Low-Medium |
| 코드 변경량 | generate/route.ts + 새 status 엔드포인트 + 프론트엔드 polling |
| Vercel 호환 | 완벽 (각 단계 < 60초) |
| 사용자 경험 | 진행률 바 + 종목별 실시간 업데이트 |

**구현 전략**:
- `Report` 모델에 `status` 필드 추가 ("pending" | "collecting" | "analyzing" | "completed" | "failed")
- `progress` 필드 추가 (0-100)
- 데이터 수집과 AI 분석을 각각 별도 API 호출로 분리
- 프론트엔드에서 3초 간격 polling

#### 방안 B: AI 분석 병렬화 (즉시 적용, A와 병행)

```
현재 (직렬):
  종목1 분석 [4초] -> 종목2 분석 [4초] -> ... -> 종목10 분석 [4초]
  요약 [3초] -> 인사이트 [3초] -> 워치포인트 [3초]
  총: ~50초

개선 (병렬):
  [종목1 | 종목2 | 종목3 | 종목4 | 종목5] 동시 -> 20초
  [종목6 | 종목7 | 종목8 | 종목9 | 종목10] 동시 -> 20초
  [요약 | 인사이트 | 워치포인트] 동시 -> 3초
  총: ~23초 (54% 단축)
```

| 항목 | 세부 |
|------|------|
| 구현 난이도 | Low |
| 효과 | 50초 -> 23초 (54% 단축) |
| 주의점 | OpenAI rate limit 고려 (분당 요청 수 제한) |
| 구현 | `Promise.allSettled`로 5건씩 배치 병렬 처리 |

#### 방안 C: Vercel Cron + 사전 생성 (중기)

```
흐름:
1. Vercel Cron (매일 09:00 KST) -> /api/cron/generate-reports
2. 모든 사용자의 관심종목 기반 보고서 사전 생성
3. 사용자 접속 시 -> 이미 생성된 보고서 즉시 로딩
4. 수동 재생성 요청 시에만 실시간 생성
```

| 항목 | 세부 |
|------|------|
| Vercel Cron 제한 | Pro: 일 1회, 최대 60초 |
| 대안 | Supabase pg_cron 또는 외부 스케줄러 (cron-job.org, 무료) |
| 적용 대상 | 관심종목 데이터 수집 (캐싱) + 분석 결과 사전 생성 |

#### 방안 D: Edge Functions + 스트리밍 (장기)

```
Vercel Edge Runtime:
- 30초 타임아웃 (Hobby), 300초 (Pro)
- Streaming 응답 지원
- 단, Prisma ORM은 Edge에서 제한적 (Prisma Accelerate 필요)
```

| 항목 | 세부 |
|------|------|
| 적용 범위 | AI 채팅 (이미 스트리밍 적합), 경량 데이터 조회 |
| 부적합 | DB 쓰기가 필요한 보고서 생성 (Prisma 호환 문제) |
| 대안 | Vercel Pro의 `maxDuration` 설정 (300초까지 확장 가능) |

### 2.3 권장 실행 계획

```
Phase 1 (즉시): 방안 B (AI 병렬화) -> 50% 시간 단축
Phase 2 (1주): 방안 A (비동기 + polling) -> 타임아웃 완전 해소
Phase 3 (2주): 방안 C (Cron 사전 생성) -> 사용자 대기 시간 0
```

| 비용 영향 | OpenAI 호출 불필요 (순수 수학 계산) |

## 5. 데이터 소스 최적화

## 6. 성능/비용 최적화

### 6.1 캐싱 전략 개선 (Critical)

**현재 문제**: `MemoryCache`가 Vercel Serverless 환경에서 cold start마다 초기화됨.

#### 해결 방안: 다층 캐싱 아키텍처

```
Layer 1: 브라우저 캐시 (React Query / SWR)
  -> staleTime: 5분, gcTime: 30분
  -> 이미 Zustand store에서 일부 처리 중이나 체계화 필요

Layer 2: Vercel Edge Cache (KV 또는 ISR)
  -> 시세 데이터: 1분 TTL (장중), 6시간 TTL (장외)
  -> 뉴스: 30분 TTL
  -> 매크로 지표: 24시간 TTL

Layer 3: Supabase (DB 캐시 테이블)
  -> 보고서 데이터: 이미 Report 테이블에 저장 중
  -> 종목 프로필: 24시간 TTL로 별도 테이블
```

#### 구현 우선순위

| 방안 | 난이도 | 효과 | 비용 |
|------|:------:|:----:|:----:|
| Vercel KV (Redis 호환) | Low | High | Pro 포함 또는 $1/월 |
| ISR (Incremental Static Regeneration) | Low | Medium | 무료 |
| Supabase 캐시 테이블 | Medium | High | 무료 (기존 DB) |

**권장**: Vercel KV를 1차 도입. ISR은 시세 페이지에 적용.

### 6.2 OpenAI 비용 절감 전략

#### (A) Batch API 적용 (보고서 생성)

```
대상: 데일리/주간 보고서의 AI 분석 (13건/보고서)
절감: 50% (Batch API 할인)
구현: 사전 생성 Cron과 연계
제한: 24시간 내 완료 보장 (실시간 생성에는 부적합)
```

#### (B) 프롬프트 최적화

| 최적화 | 현재 | 개선 | 절감률 |
|--------|------|------|:------:|
| 시스템 프롬프트 압축 | ~500 tokens | ~200 tokens (핵심만) | 10% |
| 출력 토큰 제한 강화 | max_tokens 미설정 | 용도별 최적 설정 | 15% |
| 응답 캐싱 | 없음 | 동일 종목+날짜 요청 캐싱 | 30% |
| 구조화 출력 | 자유 텍스트 | JSON mode (파싱 비용 절감) | 5% |

#### (C) 모델 선택 전략

| 용도 | 현재 모델 | 권장 모델 | 비용 변화 |
|------|----------|----------|----------|
| AI 채팅 (일반) | GPT-4o-mini | GPT-4o-mini | 유지 |
| 보고서 분석 | GPT-4o-mini | GPT-4o-mini | 유지 |
| 의도 분류 | GPT-4o-mini | GPT-4o-mini | 유지 |
| 복잡한 분석 (적정가, 시나리오) | GPT-4o-mini | GPT-4o (선택적) | +$0.05/건 |

GPT-4o-mini는 현재 최적의 선택. 특별히 깊은 분석이 필요한 엔드포인트에만 선택적으로 GPT-4o 적용.

#### (D) 비용 절감 종합 효과
